知识图谱构建流程示例（基于 #00003 会话）
================================================

前提说明
--------
1) 仅把文件手动放到 data/#00003/content 目录，不会自动触发入库。
2) 需要通过接口触发（UI 上传通常走 /api/v1/upload -> /api/v1/ingest）。
3) 下面按“上传一个 cc.txt”举例，说明完整调用链和中间产物。


一、入口阶段：文件被接收
------------------------
1) 文件上传到后端接口：
   - docthinker/server/routers/ingest.py -> @router.post("/ingest")
   - 或别名 @router.post("/upload")

2) 后端把上传文件保存到会话内容目录：
   - SessionManager.allocate_session_file_path(...)
   - 目标目录形如：data/#00003/content/cc.txt

3) 同时写入会话文件记录（用于会话文件列表）：
   - session_manager.add_document_record(...)


二、后台处理调度：按文件类型分流
-------------------------------
ingest.py 的 _background_file_processing 会把上传文件分为 4 类：
1) content_list JSON
2) image 图片
3) text 纯文本（.txt/.md）
4) complex（pdf/doc 等）

cc.txt 属于第 3 类 text，走“Plain text files”分支：
1) 读取文本：
   content = txt_path.read_text(...)
2) 先做认知增强：
   processed_text, metadata = _process_text_for_ingest(content, "file")
3) 送入 GraphCore：
   state.ingestion_service.ingest_text(processed_text, session_id=sid)
4) 更新本地认知知识图谱：
   _update_local_knowledge_graph(processed_text, metadata, session_id=sid)
5) 更新本地知识库记忆：
   _update_local_knowledge_base(content, metadata, source_type="file", session_id=sid)
6) 写 code 快照：
   _write_session_code_snapshot(... original_text=content, processed_text=processed_text, metadata=metadata)


三、processed_text 到底是什么
----------------------------
不是只传“原文”，而是：
1) 基础是原文 content；
2) 如果 cognitive_processor 可用，会追加 [Cognitive Analysis] 段，包括：
   - Summary
   - Reasoning
   - Key Points
   - Concepts
   - Entities
   - Relations
   - Inferred Relations

最终入 GraphCore 的是 processed_text（字符串）。


四、进入 GraphCore：从文本到 chunk
---------------------------------
调用链（核心）：
1) ingestion_service.ingest_text(...)
2) _insert_text -> graphcore.ainsert(text)
3) ainsert -> apipeline_enqueue_documents -> apipeline_process_enqueue_documents

在 apipeline_enqueue_documents 阶段：
1) 文档去重：基于文本内容哈希 doc-xxxx
2) 写入 full_docs（全文）和 doc_status（状态）

在 apipeline_process_enqueue_documents 阶段：
1) 分块 chunking_by_token_size：
   - chunk_token_size 默认 1200
   - overlap 默认 100
2) chunk ID = compute_mdhash_id(chunk_content, prefix="chunk-")
3) 第一阶段并行写入：
   - chunks_vdb（chunk 向量）
   - text_chunks（chunk 原文 + 元数据）
   - doc_status（处理中状态）
4) 第二阶段：实体关系抽取 _process_extract_entities


五、LLM 抽取：直接返回什么，怎么解析
------------------------------------
每个 chunk 会调用 extract_entities(...)，内部做两轮：
1) 主抽取（entity_extraction_user_prompt）
2) 补抽取/纠错（entity_continue_extraction_user_prompt，默认可 1 轮）

LLM 期望返回“纯文本行”，格式例如：
entity<|#|>Alice<|#|>person<|#|>Alice is ...
relation<|#|>Alice<|#|>Project X<|#|>collaboration<|#|>Alice works on ...
<|COMPLETE|>

解析逻辑：
1) _process_extraction_result 先按换行和 <|COMPLETE|> 切分
2) 自动修复分隔符损坏/格式异常
3) entity 行走 _handle_single_entity_extraction
4) relation 行走 _handle_single_relationship_extraction
5) 产出 maybe_nodes / maybe_edges（按 chunk 聚合）


六、缓存：LLM 原始返回如何存
---------------------------
使用 use_llm_func_with_cache：
1) 先拼 prompt + system_prompt + history，计算 args hash
2) 查 llm_response_cache（cache_type="extract"）
3) 命中则直接复用；未命中则调用 LLM，再写缓存
4) 每个 chunk 会记录 llm_cache_list（引用了哪些 cache key）

你关心的“LLM 直接返回内容”就在 llm_response_cache 的 return 字段里（原始文本）。


七、合并建图：节点和边如何落地
----------------------------
extract_entities 的 chunk 结果进入 merge_nodes_and_edges(...)：

A. 节点（实体）合并
1) 同名实体跨 chunk 合并
2) source_id（chunk 列表）合并去重
3) description 去重并排序
4) 必要时调用 LLM 做摘要合并
5) upsert 到图存储（node）
6) 同步写入 entities_vdb（实体向量）

B. 边（关系）合并
1) 边键按两端点排序后聚合（减少方向重复）
2) source_id 合并去重
3) keywords 并集去重
4) description 去重并可能摘要
5) upsert 到图存储（edge）
6) 同步写入 relationships_vdb（关系向量）


八、向量化与存储格式
-------------------
向量库实现是 NanoVectorDBStorage：
1) upsert 时批量 embedding
2) 向量会被压缩（float16 + zlib + base64）后写 JSON
3) query 时按 cosine 阈值过滤

图存储是 NetworkX + GraphML：
1) 节点边在内存图上 upsert
2) index_done_callback 时落盘 graph_*.graphml


九、最终会写到哪些文件（会话级）
-------------------------------
会话 #00003 的 GraphCore 工作目录来自 SessionManager：
rag_storage_api/sessions/#00003/

典型文件（命名规则）：
1) 图：
   graph_chunk_entity_relation.graphml
2) 向量：
   vdb_chunks.json
   vdb_entities.json
   vdb_relationships.json
3) KV：
   kv_store_full_docs.json
   kv_store_text_chunks.json
   kv_store_full_entities.json
   kv_store_full_relations.json
   kv_store_entity_chunks.json
   kv_store_relation_chunks.json
   kv_store_doc_status.json
   kv_store_llm_response_cache.json

另外还有 data/#00003/code 下的快照 JSON（用于审计/调试，不是前端主图数据源）。


十、前端图谱怎么显示
-------------------
前端知识图谱页面请求：
/api/v1/knowledge-graph/data?session_id=#00003

后端会：
1) get_session_rag(#00003)
2) 读取 graphcore.chunk_entity_relation_graph 的 get_all_nodes/get_all_edges
3) 返回 nodes/edges 给前端可视化

所以前端主图来自 GraphCore 存储，不直接从 code 快照重建。


十一、最容易混淆的点（重点）
--------------------------
1) “我把 txt 放到了 data/#00003/content”：
   - 这一步只是放文件，不等于已入库。
   - 必须触发 ingest/upload 流程，才会执行分块、LLM 抽取、建图。

2) code 快照 vs 主图：
   - code 快照：一次处理结果的静态记录（调试/回溯用）
   - 主图：GraphCore 的 graphml + vdb + kv（查询和前端展示用）
